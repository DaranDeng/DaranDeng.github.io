# 系统调用

2026年2月18日  
10:24

---

- **系统调用**：操作系统提供给应用程序的接口。
- **与库函数的区别**：
  - 库函数本质上也是要调用操作系统的系统调用，系统调用更接近底层。
  - 但库函数与系统调用的形式非常类似，都是函数形式。

### 调用层次示例
```
高级语言代码1
代码2
库函数
  （内含系统调用）
```

### 实现机制
- **机器语言层面**：使用 Trap 指令（又称陷入指令）。
- **作用**：实现中断。

### 执行流程
```
开始
  ↓
发送系统调用的请求（仍处于用户态）
  ↓
处理系统调用（处于内核态）
  ↓
返回
```

**系统调用的本质**：是用户空间与内核空间的桥梁。

**典型过程（以读硬盘为例）**：
1. 用户进程（用户态）想要读硬盘
2. 不能直接访问硬件（无权限）
3. 调用 `read()` 系统调用
4. CPU 从用户态切换到内核态
5. 内核代表用户进程执行硬件操作
6. 返回结果给用户进程
7. CPU 从内核态切换回用户态

# 中断

2026年2月15日  
10:14

---

## 1. 中断的分类

### （1）内中断（也称异常）
- 陷阱（trap）
- 故障
- 终止

### （2）外中断（也称中断）
- 时钟中断
- I/O 中断请求

---

## 2. 内外中断的区别

- **判断依据**：信号来源
  - 内中断：信号来自 CPU 内部，与当前执行的指令有关
  - 外中断：信号来自 CPU 外部
- **常见误区**：内中断就是主动放弃，外中断就是被动放弃
  - **正解**：内外中断的判断依据是信号的来源，而主动与被动要看进程是否自愿放弃 CPU，二者并无关系。
  - 外中断都是被动的
  - 内中断中：故障是被动的，trap 是主动的

---

## 3. 中断机制的基本实现原理

1. **检查中断信号**
   - 内中断：CPU 在执行指令时会检查是否有异常发生
   - 外中断：CPU 在执行完一个指令周期后检查外部的中断信号
2. **找到相应的中断处理程序**：通过中断向量表实现

---

## 4. 中断与 CPU 状态的关系

- **中断是用户态转为内核态的唯一方式**：虽然也可以通过系统调用，但系统调用本质上也是由中断实现的。

---

## 5. 开关中断

- **关中断**：禁止中断发生
- **开中断**：允许中断发生
- **特点**：开中断与关中断的指令是特权指令，必须由内核态进程才能执行。

---

## 6. CPU 状态与进程的关系

- **CPU 状态判断**：CPU 上运行的是用户态进程时，CPU 就处于用户态。
- **实现机制**：CPU 上有一个寄存器叫程序状态寄存器（PSW），其中的模式位用 0 和 1 分别代表用户态和内核态。
- **重要区分**：
  - 当 CPU 执行用户态代码时，运行的是用户写的代码
  - 当 CPU 执行内核态代码时，运行的是操作系统开发者预先写好的代码
  - CPU 处于用户态时，PSW 的模式位为 1

---

## 7. 常见误区澄清

- **误区**：进程的切换就是 CPU 状态的切换
- **正解**：错误。CPU 的状态的本质是指其上执行的代码的权限，与进程切换是两个概念。

---

## 8. 进程切换的详细过程

### 流程概览
```
用户态进程运行
    ↓
时间片用完 → 时钟中断发生
    ↓
CPU 自动从用户态切换到内核态
    ↓
执行中断处理程序 → 进程调度程序被调用
    ↓
可能切换到另一个进程
    ↓
返回用户态，新进程运行
```

### 详细步骤分解

**步骤1：进程 A 在用户态运行**
- CPU 状态：用户态
- 谁在运行：进程 A 的用户代码

**步骤2：时钟中断发生，CPU 自动切换到内核态**
- CPU 状态：用户态 → 内核态
- 谁在运行：仍然是进程 A，但执行的是内核代码
- 原因：进程 A 请求了内核服务（通过中断），内核代表进程 A 在处理这个中断

**步骤3：执行中断处理程序**
- CPU 状态：内核态
- 谁在运行：仍然是进程 A（在内核态执行中断处理）

**步骤4：调用调度程序**
- CPU 状态：内核态
- 谁在运行：仍然是进程 A（调度程序是内核的一部分）

**步骤5：调度程序决定切换到进程 B**
- CPU 状态：内核态
- 谁在运行：现在是进程 A 的内核线程在执行调度代码
- 调度程序决定："好，下一个该进程 B 了"

**步骤6：执行上下文切换**
- CPU 状态：内核态
- 谁在运行：最后一段内核代码（执行切换操作）
- 操作：保存进程 A 的上下文，加载进程 B 的上下文

**步骤7：中断返回，切换到用户态**
- CPU 状态：内核态 → 用户态
- 谁在运行：进程 B 的用户代码
- 此时，CPU 才开始执行进程 B

# 进程互斥的实现方法

2026年2月25日  
8:10

---

## 1. 硬件实现方法

### （1）关中断
- 实现方式：临界区前关中断，临界区后开中断，剩余区正常执行。
- **特点**：
  - 开关中断指令是特权指令，只能由操作系统内核进程使用。
  - **局限性**：只适用于单处理机。在多处理机环境下，每个处理器上可能都有一个进程同时访问同一段临界区代码，造成混乱。

### （2）TSL 与 swap 指令
- 本质上都是通过“是否上锁”（即 while 循环的条件判断）来确定是否可以进入临界区。
- **特点**：适用于多处理机。

### （3）软件实现方法
- 共有四种软件实现方法（具体内容略）。

---

## 2. 互斥锁

- **定义**：一个布尔型变量，用来保护临界区资源，形象地称为“锁”。
- **主要缺点**：忙等待。
  - **在单核 CPU 上**：多个进程分时共享该 CPU。当一个进程（A）在临界区中被切换出去后，另一个进程（B）被调度进来。B 因为无法进入临界区，只能在自己的时间片内“空转”（忙等待），直到时间片耗尽再切回 A。因此，**互斥锁并不适用于单核处理机**。
  - **在多核处理机中**：多个进程并发执行，如果某个 CPU 上的进程处于忙等待状态，就会持续消耗计算能力。

---

## 3. 临界区

- **定义**：临界区是指一段代码片段，在这段代码中，进程可能会访问和修改一个共享资源（例如：全局变量、文件、硬件设备等）。
- **原则**：对于同一个资源的同一段代码片段，只能被一个进程访问。

---

## 4. 信号量机制

由于上述实现方法存在缺点（如不能实现“让权等待”），因此引入信号量机制来实现进程互斥。

- **信号量**：用来表示资源的数量。
- **让权等待**：指在 CPU 内，某个进程不会一直循环等待，而是当有处于就绪队列的进程时，主动让出 CPU。
- **使用方式**：用户进程可以用一对原语对信号量进行操作（详见《进程的控制》），从而方便地实现进程互斥与同步。
- **类型**：
  - 记录型信号量
  - 整数型信号量
- P操作是wait，V操作是signal。

## 5. 实现进程互斥的代码逻辑

1. 分析并发进程的活动，划定临界区。
2. 设置信号量为 `mutex`，初始值为 1（不同的资源要设置不同的信号量），muteX就是进入临界区的名额，因此为了保证互斥，必须设置名额只有一个。
3. 在进入区执行 `P(mutex)`，申请资源。
4. 在退出区执行 `V(mutex)`，释放资源。

---

序，因此需要人为地使用技巧来控制同步。
