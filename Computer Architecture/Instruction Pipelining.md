现有几行指令（汇编代码），怎么分配到不同流水线？
如果有多条流水线（超标量处理器）
那分配指令的规则确实依赖指令间的依赖性：

text
指令1: ADD R1, R2, R3    ; R1 = R2 + R3
指令2: SUB R4, R1, R5    ; R4 = R1 - R5  （依赖R1）
指令3: MUL R6, R7, R8    ; R6 = R7 × R8  （无依赖）
分配策略：

有依赖的指令（指令1和指令2）→ 不能同时发射到不同流水线，必须等指令1写回后才能执行指令2

无依赖的指令（指令1和指令3）→ 可以同时发射到两条流水线并行执行

这就是指令级并行（ILP） 的核心。

in-order	指令按照程序顺序（即编译后的指令流顺序）依次进入执行单元，完成后再提交结果。如果某条指令 stall，后面的指令即使可以执行也必须等待。
Out-of-order	指令可以不按程序顺序执行。只要操作数准备好了、执行单元空闲，就可以提前执行，结果再按原顺序提交。

概念	含义	由谁决定
program order	指令在程序中的先后顺序	程序员/编译器
data dependency order	数据必须被“先写后读”的顺序	数据本身的流动

三种数据依赖类型：
名字依赖：WAW,WAR
真依赖：RAW。
显然，名字依赖是假依赖，因为只是寄存器名字冲突和寄存器不够用，不是数据真的需要传递。这破坏了乱序执行的高效性，那么怎么解决呢？

一、Front-End（前端）
作用：负责取指令、译码、预测分支，保证指令能持续供给到后端。

主要包含的阶段：

指令预取（Instruction Fetch）

指令译码（Instruction Decode）

分支预测（Branch Prediction）

指令缓存（Instruction Cache）

关键点：

前端的目标是“喂饱”后端

如果前端预测错误（分支猜错），就会导致流水线冲刷（flush）

二、Back-End（后端）
作用：负责执行指令、处理数据、写回结果。

主要包含的阶段：

操作数读取（Operand Fetch）

指令执行（Execute）

写回结果（Write Back）

数据缓存（Data Cache）

关键点：

后端关注的是数据依赖、执行单元调度

如果后端出现数据冒险（RAW等），也会导致停顿
