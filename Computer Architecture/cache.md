# 缓存

## 解决什么问题？
采用传统的把主存和CPU分开的冯诺依曼结构，require more clock cycles to read data from CPU than execute。  
所以把一些需要经常访问的数据存储在离CPU更近的缓存，减少CPU等待时间。

## Why we need multi-level memory
For practical and economic reasons：
- 更快的内存更贵、容量小
- 更慢的内存更便宜、容量大
- 多级结构在“速度”与“成本/容量”之间取得平衡

## even caches have multiple levels. Why？
- L1 cache：最快、最小、最贵（通常几十KB），离CPU最近
- L2 / L3 cache：稍慢、更大、更便宜（几MB到几十MB）
- 多级缓存可以：
  - 让**最频繁访问的数据**留在L1
  - 较常用但L1放不下的数据放在L2/L3
  - 大部分访问在最快的小缓存中命中 → 整体性能接近L1，容量接近L3

> 📌 注意：L1通常还分为 **L1 i-cache（指令缓存）** 和 **L1 d-cache（数据缓存）**

## How can we make sure to hit in cache as often as possible？
**局部性原理（Locality of Reference）**

1. **时间局部性**  
   刚用过的数据很可能马上再用  
   → 放到缓存中，避免重复访问主存

2. **空间局部性**  
   用到一个数据时，它周围的数据也很可能被用到  
   → 一次读入一个**块（block）**，而不只是单个字节

# 缓存映射（Cache Mapping）

## 问题1：什么是缓存映射？为什么需要它？
当CPU需要读取某个数，发现缓存没有。因此从主存直接加载过来，顺便拷贝一份到缓存。  
因为主存（$\ge 1$GB）远大于缓存（$\le 1$MB），所以必须用地址信息来管理。

## 问题2：直接映射（Direct Mapping）如何计算缓存组号？
公式：  
$$ \text{cache set号} = \text{主存块号} \mod \text{缓存总set数} $$

- 每个 set 中只有 **1 个块**
- 多个主存块会映射到同一个 set → 冲突、频繁替换、命中率低

**例题**：如果缓存有 8 个 set，主存块号 0、8、16 会映射到同一个 set 吗？为什么？  
✅ 会，因为 $0 \mod 8 = 0$，$8 \mod 8 = 0$，$16 \mod 8 = 0$。

**Tag 存什么？**  
$$ \text{tag} = \lfloor \frac{\text{主存块号}}{\text{缓存总set数}} \rfloor $$

## 问题3：完全相联映射（Fully-Associative Mapping）的优缺点是什么？
- **优点**：任何主存块可以放在任意缓存行 → 几乎无冲突
- **缺点**：
  - 需要存储**完整地址（tag）**，硬件开销大
  - 查找时必须**并行比较所有缓存行**，硬件复杂、功耗高

## 问题4：N路组相联映射（N-way Set-Associative）如何结合前两者的优点？
- 用 **直接映射** 方式确定 **组（set）**
- 组内采用 **完全相联** 方式放置（N 个 way）
- CPU先给了一个地址，先直接用取模的方式找到组号，再到组内随机找。因为先按照直接映射的方式映射到组，第一遍每个组都只存一个块，之后才是存第二个块
- 减少冲突，硬件复杂度适中

## 问题5：如何计算地址中的 Byte 偏移、Set 索引和 Tag？
以课件例子为例：  
- 2-way set-associative  
- 256 个 set  
- 每行（cache line）= 16 bytes  
- 32 位地址

**计算过程**：
- Byte 偏移：$\log_2(16) = 4$ bit
- Set 索引：$\log_2(256) = 8$ bit
- Tag：$32 - 4 - 8 = 20$ bit

地址划分格式：  
$$ \{\text{Tag},\ \text{Set},\ \text{Byte}\} $$

**例题**：地址 $FFFFFFE4$（十六进制）如何划分？  
- 二进制：`1111 1111 1111 1111 1111 1111 1110 0100`  
- Tag = `FFFFF`（20 bit）  
- Set = `FE`（8 bit）  
- Byte = `4`（4 bit）

## 问题6：三种映射方式中，哪一种在实际中最常用？
**N路组相联映射**（通常 2-way 到 16-way）。  
它平衡了直接映射的简单性和完全相联的灵活性。
## 关键术语
- **cache hit**：要访问的数据已在缓存中
- **cache miss**：数据不在缓存中，需从主存读取
- **miss 后动作**：从主存读数据，并**拷贝一份到缓存**

# Cache updating and coherence

## 主存远大于缓存（<1%），怎么映射？
映射：把主存的数据或指令以块的形式传输到对应编号的缓存块。

## 为什么物理上实际需要的存储空间 > 标称的数据容量？
因为除了数据还有tag,LRU 位。缓存标称大小 ≠ 物理晶体管开销。
## 在组相联缓存中，地址怎么决定具体放到哪里？
+ 先是按照固定的地址决定放到哪个组
+ 然后在运行时才会决定放到这个组的哪个块。一开始用组里第一个空 way。
当组里所有 way 都满了，必须覆盖某个已有数据 → 这时才需要替换算法（LRU、随机等）。

## 替换算法
+ 触发条件：当CPU需要读取某个数，发现缓存没有。因此从主存直接加载过来，顺便拷贝一份到缓存。此时若缓存所有路都满了，采用替换算法。
+ 三种策略，比如在统计窗口里访问频率最低，也就是自程序执行开始以来的总访问次数，LFU。自从上次访问以来时间最少的，LRU。还有：FIFO。最高效：LRU
+ 注意LRU算法的实现，用到housekeeping” bits。在 2-way组相联 中，每个组的1 位足够表示哪个 way 更久没被访问。这个位是运行时动态更新的。

## 计算地址中的 Byte 偏移、Set 索引和 Tag和计算物理上实际需要的存储空间有点混淆？
后者是指实际存储的数据信息所需要的空间。前者是用来查找的信息，不需要存储在元件里面，是查找是哪个缓存用的，主要体现在CPU发地址这个过程，硬要说就是存储在bus里面。计算题要注意。

## update
+ 跟替换是两码事，这个触发条件是每次写入缓存时会造成主存与缓存内容不一致。
+ 解决思路？
+ write-through，每当我们执行一次写操作，我们会立刻将数据写入到更低层级的存储器中。由于我们的存储系统是包含性 (Inclusive) 的，我们可以保证在更低层级也一定能找到这个块（因为L1有的，L2/L3也有）。还要注意，我们通常不会一次性写入一个64字节块的全部字节。相反，我们以更细的粒度进行写入——每次写入2到8个字节。这些小的写入操作会被发送到更低层级的缓存，在那里对应的数据块会被更新
+ write-back。并不会自动写入更低层级的缓存。每条缓存行存储一个修改位:if 1 ，说明这个缓存块被修改过，此时并未同步给低层级的，替换操作会有额外的一个操作，那就是将旧的即将被扔掉的块内容写回到低层级的，再替换掉；if 0，它可以被无缝地驱逐。对缓存，第一次写操作就造成不一致了。不急着对主存写。而是当该缓存块要被替换出去（evict）时，如果它是“脏的”（dirty，即被写过且与主存不一致），才将其写回主存。因为第二次写入缓存不见得会覆盖原来的内容，可能是写到其他的块,对于存储原来内容的缓存块不急着保存到主存里面。
+ 举个场景帮理解（把概念串起来）:

系统为 Inclusive + write-back：
L1 写一行 → L1 置 dirty，但 L2 必须也有该行（包含性），所以要么 L1 同步更新 L2（在某些实现中通过写合并/转发），要么在 L1 驱逐时写回到 L2；同时，若其他核想读该行，MESI/总线会触发 L1 将 Modified 行写回并转为 Shared/Invalid 等。
系统为 Exclusive + write-back：
L1 写一行 → L2 没有该行（除非是被驱逐来的），因此不需要同步 L2；当 L1 驱逐，数据会移动到 L2（而不是写回主存），这能减少重复拷贝

+ 两种策略的各自优缺点：在一个具有高度时间局部性 (Temporal Locality) 和大量写操作的系统里，写直达缓存显然是低效的。第一，由于时间局部性，我们预期同一个块会被反复访问。第二，由于某些工作负载有大量的写操作，我们需要向更低层级写入大量的数据块——只要它们在高层级被修改一次，就得写一次。这些写操作是不必要的，如果使用写回缓存则可以避免。但另一方面，写直达缓存简单且支持无缝驱逐。


## lookup查找
Lookup查找操作，CPU给定了一个memory address，那么cache会看自己里面是否存储这个地址对应的块，随后返回击中/未击中信号；若未击中，那就一级一级到更低层次的缓存获取块的内容，直到主存。一旦有击中信号，就会把块的内容覆盖到所有层级的缓存块中。因为要保证inclusive。
